# Life Evolve Updater - 中文

## 标准文件

- `final.md`：长期战略和当前最终判断。
- `bridge.md`：`final.md` 与 `donelist.md` 之间的执行桥，把长期判断翻译成已有资产支撑的商业路线、执行优先级、验证指标和近期行动。深度创建或重写时使用 `life-evolve-bridge`。
- `frontier.md`：前沿机会、外部地图、技术/数据杠杆点。
- `donelist.md`：有证据支撑的已完成工作清单，每个项目必须有完成度百分比。
- `memory.md`：人生时间线、长期事实、生平经历复原。
- `soul.md`：性格、价值观、决策规则、操作约束。
- `hobbies.md`：可能发展成副业的严肃兴趣。
- `progress.md`：仓库本身和更新系统的完成度。

## 原生中文 Prompt

更新中文文件时，直接用中文思考和写作：

```text
从证据更新这个 life-evolve 文件夹。
保留已核实事实，推断标为“推断”。
保持八个文件一致。
保留人物固有的事实、经历与判断。
把已经交付的项目写入带完成度的 donelist。
评估兴趣发展为副业的可能性。
把生平经历整理成服务未来决策的操作记忆。
```

## 工作流

1. 先读中文八个文件。
2. 如果是多语言仓库，确认中文是否是 source-of-record，并同步英文/日文版本的逻辑。
3. 编辑前先收集证据：
   - 本地 Git 仓库和最近提交；
   - README / package metadata；
   - 本地 memory 文件和笔记；
   - 用户明确给出的产品状态；
   - 只有外部状态会变化时才查公开网页。
4. 每次变更同时核对八个文件的关联项。
5. 当 `final.md` 或 `donelist.md` 变化时，检查 `bridge.md` 是否需要新增、推进、降级、hold 或 reject 对应桥接条目。
6. `donelist.md` 里每个主要项目或方向都写明完成度、当前证据和距离 100% 的闭环条件。
7. `progress.md` 里区分“仓库完成度”和“项目完成度”。
8. commit 前做隐私检查。

## 隐私规则

- public repo 只收录可公开、去标识且有证据的摘要。原始聊天、密钥、credential、私有数据库和敏感第三方信息留在私有环境。
- private repo 收录任务所需的证据摘要。raw dump 留在原始证据位置。
- private skill 同步到 public skill repo 前，移除人名、私密经历、私有 URL、repo 扫描和证据文件。

## 完成度规则

- 0-20%：想法、研究、占位或早期笔记。
- 21-40%：有原型或规划 repo。
- 41-60%：已有可运行实现；下一闭环是产品化。
- 61-80%：产品或系统的主体已经建成；下一闭环是运营、用户、包装或 hardening。
- 81-95%：已经上线或接近生产；下一闭环是运营、规模、商业化或 polish。
- 96-100%：有用户、收入、留存、维护闭环，或 artifact 本身已刻意完成。

## 写作规范

- 每节首句直接写该文件的事实、判断、证据或动作。
- 每句承载一项事实、推断或动作。
- 保留专名、日期、路径、commit、百分比、状态值和人物辨识度。
- 事实绑定证据。真实推断统一标为“推断：”。缺少证据基础或证据冲突未解的关系断言整句删除，不留缓冲词或占位词。冲突本身影响决策时，逐项写明来源事实与差异，不补关系判断。
- 每项完成度同时绑定当前证据与下一闭环条件。
- 有明确篇幅目标时，正文落在目标的 90%–110%；没有目标时，证据决定最短完整篇幅。
- 正文省去检索过程、核验过程、泛化免责声明、边界铺垫、复述、总结套话、情绪填充、叠加限定和空占位。隐私、安全或法律边界改变可公开内容或可执行动作时，直接写具体约束与许可路径。
- 省去“值得注意的是”“需要指出的是”“综上”“换言之”“作为AI”“作为一个AI”“模型认为”“事实上”“实际上”“基本上”“简单来说”“显然”“暂无”“未见”“从缺”“待考”。正向事实句取代防御式转折。
- 交付前静默核对禁用语、重复限定、无证据断言、八个文件、数字、隐私和三语逻辑。输出中省去检查过程。

## 输出标准

报告：

- 改了哪些文件；
- 用了哪些证据；
- 哪些完成度发生变化；
- 哪些事实是推断；
- repo visibility 和 push 状态；
- 有证据支撑的剩余 gap。
