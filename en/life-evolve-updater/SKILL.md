# Life Evolve Updater - English

## Canonical Files

- `final.md`: long-horizon strategy and current thesis.
- `bridge.md`: execution bridge between `final.md` and `donelist.md`; translates long-term theses into asset-backed commercial routes, execution priorities, validation metrics, and near-term actions. Use `life-evolve-bridge` for deep creation or rewrites.
- `frontier.md`: frontier opportunities, external maps, and leverage points.
- `donelist.md`: evidence-backed work record with per-project completion percentages.
- `memory.md`: life timeline, durable facts, and biographical reconstruction.
- `soul.md`: temperament, values, decision rules, and operating constraints.
- `hobbies.md`: serious interests that may become side businesses.
- `progress.md`: repository and update-system completion tracking.

## Native Prompt

When updating English files, think and write directly in English:

```text
Update this life-evolve folder from evidence.
Preserve verified facts and label inference.
Keep the eight files consistent.
Preserve the person's specific facts, history, and judgments.
Record shipped work in a done-list with completion percentages.
Assess interests for hobby-to-business optionality.
Convert personal history into operational memory for future decisions.
```

## Workflow

1. Read all eight English files first.
2. If this is a multilingual repo, compare against the source-of-record language folder.
3. Gather evidence before editing:
   - Git repositories and recent commits;
   - README/package metadata;
   - local memory files and notes;
   - user-provided product status;
   - public pages only when current external status matters.
4. Reconcile every related field across all eight files in each update.
5. When `final.md` or `donelist.md` changes, check whether `bridge.md` needs a new, promoted, downgraded, held, or rejected bridge item.
6. In `donelist.md`, give every major project or direction a completion percentage, current evidence, and the closure condition separating it from 100%.
7. In `progress.md`, track repository maturity separately from project maturity.
8. Run a privacy check before committing.

## Privacy

Public repositories contain sanitized, publishable, evidence-backed summaries. Raw chats, secrets, credentials, private database dumps, and sensitive third-party data remain in private storage. Private repositories may contain evidence-backed summaries required for the work.

## Completion Percentages

Use this scale:

- 0-20%: idea, research, placeholder, or early notes.
- 21-40%: prototype or planning repo exists.
- 41-60%: a working implementation exists; product readiness is the next closure condition.
- 61-80%: the product or system core exists; operation, users, packaging, or hardening form the next closure condition.
- 81-95%: launched or near production; operation, scale, monetization, or polish form the next closure condition.
- 96-100%: closed loop with users, revenue, retention, and maintenance, or intentionally complete artifact.

## Writing Standard

- Open each section with its fact, judgment, evidence, or action.
- Give each sentence one fact, inference, or action.
- Preserve names, dates, paths, commit ids, percentages, status values, and distinguishing personal details.
- Bind factual claims to evidence. Prefix genuine inference with `Inference:`. Omit each unsupported relationship or claim as a whole sentence. When unresolved conflicting evidence changes a decision, state each sourced fact and its difference without adding a connecting judgment.
- Bind each completion percentage to current evidence and its next closure condition.
- When the user supplies a target length, keep the delivered body within plus or minus 10 percent. With no target, let evidence determine the shortest complete section.
- Omit research narration, verification narration, generic disclaimers, boundary preambles, repeated conclusions, empty placeholders, affective filler, and phrases such as “it is worth noting,” “in summary,” “in other words,” “as an AI,” “actually,” “basically,” “simply,” and “clearly.” State a privacy, safety, or legal boundary only when it changes publishable content or the available action; name the exact constraint and permitted path.
- Before delivery, silently check boilerplate, repeated qualification, unsupported claims, all eight files, numbers, privacy, and multilingual logic. Keep the check itself out of the output.

## Output Standard

Report changed files, evidence sources, completion percentage changes, inferred facts, repository visibility, push status, and evidence-backed remaining gaps.
